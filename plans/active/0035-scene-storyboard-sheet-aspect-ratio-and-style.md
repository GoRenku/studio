# 0035 Scene Storyboard Sheet Media Producer Boundary And Style

Date: 2026-05-30

Status: proposed

## Goal

Fix Scene Storyboard Sheet generation by putting the workflow in the right
skill and making its crop rules match Location Environment Sheets.

There are two separate skills at play:

- `scene-shot-designer` identifies, designs, validates, persists, and activates
  Scene Shot Lists.
- `media-producer` generates and imports media for a selected purpose, including
  `scene.storyboard-sheet`.

The current skill shape is wrong because `scene-shot-designer` owns too much of
the storyboard image workflow. Shot design and media production should not be
lumped together. Once a shot list exists, storyboard sheet generation should
follow the same media-purpose pattern as `location.environment-sheet`.

This plan fixes:

- the skill boundary;
- the storyboard sheet aspect ratio;
- the per-shot panel aspect ratio;
- the overcrowded ten-shot sheet problem;
- the bad cinematic black-and-white aesthetic;
- the blind grid slicing failure;
- the project aspect-ratio propagation bug.

## Current Problem

### Wrong Skill Ownership

`scene-shot-designer` currently says it can optionally generate one storyboard
sheet and slice it into per-shot images. That does not match the shape of the
newer media workflows.

The correct pattern already exists in `media-producer`:

1. Resolve the project, target purpose, and target id.
2. Read generation context for that purpose.
3. List model choices for that purpose.
4. Create a persisted media generation spec.
5. Estimate cost.
6. Run only after approval.
7. Inspect generated images before import.
8. Import finished files separately.

Storyboard sheets are media. They should be generated by `media-producer` as a
purpose-specific workflow, not by the shot-list authoring skill.

### Blind Slicing Failure

The old storyboard helper:

```text
/Users/keremk/Projects/aitinkerbox/studio-skills/skills/scene-shot-designer/scripts/slice_storyboard_grid.py
```

uses Pillow to divide the whole sheet into equal cells. It does not use vision,
does not locate the actual generated panels, does not exclude gutters or labels,
and does not enforce the intended per-shot aspect ratio.

That caused the `urban-basilica` failure:

```text
Generated sheet: 1088 x 608
Grid used:       5 columns x 2 rows
Imported slices: approximately 217 x 304 and 220 x 304
Expected slices: landscape 16:9
```

The generated images were not merely imperfect crops. They were the wrong
shape, with inaccurate borders, because the workflow trusted rough grid math
instead of visual inspection.

### Missing Vision Requirement

Location Environment Sheets already state the right rule:

```text
Inspect the returned composite, use vision to identify the four scenic image
blocks, crop those four blocks, and inspect the four slices.
```

Storyboard Sheet generation needs the same rule, explicitly and without
fallbacks:

```text
Inspect the returned composite, use vision to identify the actual storyboard
panel image blocks for the selected shots, crop those blocks, and inspect every
slice before import.
```

The crop boxes must be selected by vision for the specific returned image. They
must not come from fixed coordinates, border detection, marker detection, OCR,
rough quadrants, or a reusable grid slicer.

### Aspect Ratio Confusion

Scene Storyboard Sheet generation currently treats one `imageFrame` value as if
it can describe both the full composite sheet and the individual shot panels.
Those are different surfaces:

- the composite sheet should be a readable reference document, normally `4:3`;
- the storyboard panels inside the sheet should match the project frame,
  normally `16:9` for a movie project;
- the imported per-shot files should preserve that project frame.

There is also an aspect-ratio propagation bug. The Studio Project Details UI can
show `16:9` because the form falls back to that value when the stored project
aspect ratio is missing. Generation context can still receive `null`, so a spec
with `imageFrame: "project"` fails validation and agents fall back manually.

### Sheet Is Too Crowded

Putting ten shots on one sheet is too dense. It encourages cramped drawings,
weak prompt adherence, and bad crops. The accepted default for this plan is:

```text
maxShotsPerSheet = 4
```

For a ten-shot scene, `media-producer` should create three storyboard sheet
specs:

```text
Sheet 1: shots 1-4
Sheet 2: shots 5-8
Sheet 3: shots 9-10
```

### Wrong Aesthetic

The bad generated sheet looked like a cinematic black-and-white contact sheet.
The desired direction is closer to the attached reference image:

- hand-drawn graphite or light ink;
- warm off-white paper;
- subtle gray or beige wash;
- clear landscape storyboard cards;
- dark gutters or a considered dark presentation surround;
- production-board clarity, not final film still polish.

## Correct Skill Boundary

### `scene-shot-designer`

`scene-shot-designer` owns:

- resolving the current project and target scene;
- reading screenplay scene context;
- reading cast, location, project aspect ratio, and active Lookbook guidance;
- designing shot coverage;
- validating and persisting Scene Shot List JSON;
- activating or restoring Scene Shot Lists;
- handing off to media generation when the user asks for storyboard images.

`scene-shot-designer` must not own:

- media generation model selection;
- persisted media generation specs;
- paid generation estimates or approval tokens;
- provider prompt structure for storyboard sheets;
- storyboard sheet visual style instructions;
- image cropping or import;
- storyboard sheet samples;
- crop scripts.

When a user asks for storyboard images while using `scene-shot-designer`, the
skill should ensure a shot list exists, then hand off to the media generation
workflow:

```text
Use media-producer with purpose `scene.storyboard-sheet`, target
`scene:<scene-id>`, and the active `shotListId`.
```

This is a skill-level handoff, not a duplicated workflow.

### `media-producer`

`media-producer` owns all `scene.storyboard-sheet` media work:

- reading generation context;
- listing model choices;
- choosing or honoring the user-selected model;
- creating one or more persisted generation specs;
- estimating cost;
- running generation after approval;
- inspecting the returned composite sheets;
- using vision to identify actual storyboard panels;
- cropping only the actual panel image blocks;
- inspecting every slice;
- importing the composite sheet and explicit shot files.

This should mirror `location.environment-sheet`, with scene shot panels instead
of azimuth views.

## Generation Spec Contract

Extend the Scene Storyboard Sheet spec so the sheet frame and shot frame are
separate:

```ts
{
  purpose: 'scene.storyboard-sheet',
  target: {
    kind: 'scene',
    id: string
  },
  modelChoice: string,
  prompt: string,
  shotListId: string,
  sheetFrame: '4:3' | string,
  shotFrame: 'project' | '16:9' | string,
  shotIds: string[],
  takeCount: 1,
  seed?: number | null,
  detail?: 'standard' | 'high' | string,
  outputFormat?: 'png' | 'jpeg' | 'webp',
  title: string
}
```

Rules:

- `shotListId` is required.
- `shotIds` is required.
- `shotIds.length` must be between `1` and `4`.
- every `shotId` must belong to `shotListId`;
- `sheetFrame` defaults to `4:3`;
- `shotFrame` defaults to `project`;
- `shotFrame: "project"` resolves to the effective project aspect ratio;
- movie projects without a stored aspect ratio resolve to `16:9`;
- provider payload aspect ratio comes from `sheetFrame`, not `shotFrame`;
- prompt construction must name the resolved shot frame, such as "each panel is
  a clean 16:9 landscape storyboard frame."

Remove `imageFrame` for this purpose and update callers directly to
`sheetFrame` and `shotFrame`.

## Scene Storyboard Sheet Reference

Create `media-producer/references/scene-storyboard-sheet.md` using the same
shape as `references/location-environment-sheet.md`.

Required sections:

- Purpose key and target format.
- Required workflow.
- Prompt inputs.
- Sheet layout.
- Historical guardrails.
- Vision-guided cropping expectations.
- Import expectations.

The required workflow should explicitly say:

```text
1. Read context with `renku generation context --purpose scene.storyboard-sheet --target scene:<scene-id> --shot-list <shot-list-id> --json`.
2. List model choices with `renku generation model list --purpose scene.storyboard-sheet --target scene:<scene-id> --shot-list <shot-list-id> --json` unless the user already chose one.
3. Verify the target Scene and Shot List exist.
4. Split the selected shots into batches of at most four shots.
5. Create one persisted spec per batch and estimate cost before any paid run.
6. Run only after user cost approval for the bound provider request.
7. Inspect each returned composite, use vision to identify the actual storyboard panel image blocks, crop those blocks, and inspect every slice.
8. Import only when the sheet and all slices are clean, useful, and match the resolved shot aspect ratio.
```

The Vision requirement should be as explicit as the location skill:

```text
Renku generates one provider image for each storyboard sheet. The agent then
uses vision to locate the actual storyboard panel image blocks for the selected
shots. Crop only those image blocks.

Crop around the image content, not the panel frame. Exclude labels, gutters,
decorative sheet background, borders, debug marks, and any non-shot content
unless the user explicitly asks to preserve borders. The crop boxes are selected
by vision for the specific returned image; they are not fixed coordinates and
are not discovered through border detection, marker detection, OCR, rough
quadrants, or grid slicing.

Do not import automatically when the image no longer provides clean useful
storyboard panels for the selected shots. Show the composite to the user,
explain that the generation is not good enough for a storyboard sheet, and ask
whether to regenerate.
```

## Prompt Guidance

Storyboard sheet prompts should use the active Lookbook and the shot list, but
they should render as hand-drawn production storyboards, not final film stills.

Suggested base instruction:

```text
Create one 4:3 storyboard sheet as a single finished image. Arrange up to four
clean 16:9 landscape storyboard panels in shot-list order. Use hand-drawn
graphite pencil and light ink on warm off-white paper, with subtle gray/beige
washes, visible sketch construction, readable staging, and production-board
clarity. Use rounded white panel cards with dark gutters or a considered dark
presentation surround. Preserve the project's historical visual language and
shot intent, but render the frames as storyboard drawings rather than cinematic
black-and-white film stills. Keep labels in the margin or sheet header; do not
place labels, debug marks, crop marks, or decorative text inside the shot image
content.
```

Prompts should include:

- the selected shot IDs and shot labels for that sheet only;
- each selected shot's subject, framing, camera angle, movement, and story
  purpose;
- referenced cast and locations;
- the active Lookbook's relevant palette, texture, lighting, and composition
  notes;
- the resolved per-shot aspect ratio.

Prompts should avoid:

- photorealism;
- final film still polish;
- heavy charcoal/noir contrast;
- cinematic black-and-white grading;
- ten-panel contact sheets;
- labels inside panel image content;
- crop marks, red dots, debug overlays, and template artifacts.

## Import Contract

Storyboard sheet import should work like location sheet import: the agent imports
one composite and explicit already-cropped files.

For each generated sheet:

- one `composite` sheet file is required;
- one cropped file is required for every `shotId` in that sheet spec;
- duplicate `shotId`s in one manifest are invalid;
- every imported `shotId` must belong to the target `shotListId`;
- the import no longer needs to cover every shot in the full shot list;
- all files must be project-relative and inside the project;
- source-only import is not valid for this purpose.

This allows a ten-shot list to be represented by three storyboard sheet imports:

```text
Sheet 1 import: composite + shots 1-4
Sheet 2 import: composite + shots 5-8
Sheet 3 import: composite + shots 9-10
```

Studio should store the imported files and shot relationships. It should not
store crop boxes, extraction confidence, grid layout, crop diagnostics, or other
agent-side crop mechanics.

## Acceptance Criteria

- `scene-shot-designer` no longer owns storyboard media generation.
- `media-producer` has a first-class `scene.storyboard-sheet` purpose section.
- `media-producer/references/scene-storyboard-sheet.md` mirrors the structure
  and explicit Vision language of `location-environment-sheet.md`.
- Storyboard generation uses `sheetFrame: "4:3"` for the composite.
- Storyboard panels use the resolved project aspect ratio, normally `16:9`.
- One sheet contains at most four shots.
- Longer shot lists produce multiple specs and imports.
- Cropping is explicitly vision-guided.
- Rough grid slicing is explicitly disallowed.
- `slice_storyboard_grid.py` is deleted.
- No replacement crop script is added.
- Storyboard sheets visually resemble hand-drawn production boards, not
  cinematic black-and-white contact sheets.
