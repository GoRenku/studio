Plan 0042 visual explainer

Generation plans are dependency graphs.

The AI Production pane should not ask, "How much does this one video model call cost?" It should ask, "What complete graph of assets and generations is needed to make this take, and what does that graph cost?"

The short version

Model variants live in an engine-owned route catalog. Project references live in a core dependency map. Studio renders both, but owns neither.

01 - The forest

Two facts meet in core: model truth and project truth.

Engines knows provider reality. Core knows project reality. Studio should only display the answer core produces.

The important shift: the browser no longer estimates a single provider call. It asks core for a project-aware plan.
Engine concept

Model family

The user-facing base model row, such as Kling 3.0 or Veo 3.1.

Engine concept

Route

One family plus one intent maps to one provider endpoint, one schema, one settings list, and one pricing function.

Core concept

Dependency map

The graph of reused assets, missing inputs, generated dependencies, and the final generation.

UI concept

Plan projection

The compact table rows, run controls, estimate total, and dialog details rendered from the core plan.

02 - Models and variants

The generic part is the route shape, not the parameters.

Different models can have wildly different provider variants. The system handles that by making every intent route explicit.

What a route answers

  • Does this base model support this intent?
  • Which exact provider model id is used?
  • What inputs does that provider route need?
  • Which settings can be shown in Run Setup?
  • How should duration be displayed and encoded?
  • Can this route be priced?
Why schemas alone are not enough

Provider schemas can validate a route, but they cannot tell us product semantics like "this is the Kling 3.0 first-frame route" or "this base row should be grayed out for first-and-last-frame."

A base row can stay clean while each route remains precise.
Question Answered by Example Why it matters
What should the table row show? Model family Seedance 2.0 Users choose base models, not provider endpoint names.
Is first-frame supported? Route catalog Kling 3.0 + first-frame -> image-to-video route Capability is explicit and reviewable.
Should Run Setup show seed? Selected route parameters Veo image route does not expose seed Unsupported fields cannot leak into payloads.
How should duration render? Route duration domain continuous: 3-10s, discrete: 4, 8, 12s Display and payload encoding come from the same source.
Can it estimate? Pricing registry Ready, No price, Needs input The UI shows missing pricing without disabling override runs.

03 - Adding a model

Adding a model now means adding provider facts plus product routes.

The old work still exists: add the provider schema and register it. The new work is declaring how that provider model appears in the Studio product.

1. Add provider schema Add the fal.ai model schema and register the provider model in the provider catalog.
2. Add family row Declare the user-facing base model label, version, provider, and stable model choice id.
3. Add routes For each supported intent, map to the exact provider model, inputs, settings, duration, and defaults.
4. Add pricing Register or select the pricing function. If pricing is missing, the route is marked No price and needs override before a run.
5. Validate route matrix Validation checks schemas, required fields, durations, defaults, and estimateability.
So yes, adding a model is more work now.

That extra work is the product contract. The schema tells us what the provider accepts. The route catalog tells Studio what the model means, when it is selectable, what controls it should show, and whether the selected route can really estimate and run.

04 - Dependency map

The shared service is a graph builder, not a video-only hack.

A generation request becomes a graph of assets and generation steps. The same graph powers the estimate, the Preview Take Plan dialog and the agent run.

The dependency map is the forest. The selected model route is only one node in that forest.
Resolver

Find existing assets

If a selected first frame, character sheet, location sheet, or lookbook already exists, the map adds it as a reused asset.

Planner

Plan missing assets

If an input is missing and the product says it can be generated, the map adds a dependency generation line.

Estimator

Sum honest totals

The total is complete only when every required line is priced and no required input is missing.

05 - Dependency execution

Core resolves the tree; the agent executes the levels.

A plan is not just a flat list of missing inputs. Core walks dependencies recursively, checks that the graph is valid, then returns the order in which work can safely happen.

1. Start at final video The selected route declares the immediate slots it needs: first frame, reference images, audio input, or another route input.
2. Resolve each slot Core first looks for a selected project asset. If none exists, the slot either plans a concrete generation purpose or asks for an attachment.
3. Recurse into planned nodes A planned character sheet can depend on a lookbook. A planned location sheet can do the same. The resolver keeps walking until it reaches leaves.
4. Validate the graph Cycles, missing purpose contracts, invalid route slots, and unsupported provider mappings become structured diagnostics.
5. Return work levels Core returns a topological order, then groups independent nodes into levels the agent can work through.

  "execution": {
    "topologicalNodeIds": [

      "character_urban_sheet",
      "location_basilica_sheet",

      "final_video"
    ],
    "levels": [
      ["lookbook_reference"],
      ["character_urban_sheet", "location_basilica_sheet"],

}

Transitive dependencies

The final video may need a character sheet. That character sheet may need a lookbook reference. The plan should show both levels, not pretend the character sheet is a leaf.

Cycle diagnostics

If purpose declarations accidentally loop, core reports the exact path. The affected branch is unavailable until the contract is fixed.

Topological order

Every node appears after its prerequisites. That gives the agent one safe serial sequence even before it decides what can run in parallel.

Execution levels

Level 1 must finish before level 2. Inside a level, independent nodes can be delegated to workers or subagents because they do not need each other's outputs.

Concept What core returns What the agent does Why it matters
Existing asset A ready node with asset and file ids. Uses the file as an input; no generation needed. Reuse stays explicit and costs $0.
Planned generation A node with a concrete purpose and draft spec. Runs the normal media flow for that purpose, then imports the result. Each asset keeps its own purpose contract.
Required attachment A missing node with media kind and slot id. Asks the user to attach or upload a matching file. Audio or video inputs can exist before generation support exists.
Unpriced line A planned node with an unpriced reason. Requests explicit unpriced override before a real run. Missing estimates are visible without preventing development runs.
Topological order A serial node list with prerequisites first. Uses it as the safest fallback execution sequence. The final video cannot run before its required inputs exist.
Cycle A diagnostic with the dependency path. Stops that branch and fixes the declaration before running. Bad contracts fail clearly instead of recursing forever.
The agent does not trust stale placeholders.

After each execution level completes, generated outputs must be imported or attached as project assets. The agent then asks core for a fresh plan, so the final video receives real file references rather than imagined future outputs.

06 - Reference intent

Reference intent resolves a dynamic bundle from the shot.

Reference does not mean one hard-coded reference image. It means "collect the references that matter for this shot and this route."

Reference slot Where it comes from Cardinality If asset exists If missing
Character sheets Cast members present in the shot zero to many reuse generate or attach
Location sheets Locations attached to the scene or shot zero to many reuse generate or attach
Lookbook sheets Active visual language selection zero to many reuse generate or attach
Audio references Only routes that accept audio inputs zero or one, or many by route reuse attach or generate

No future-only rows

When a new reference domain exists, it joins through the same asset selector, purpose, and dependency-slot contract. Studio should not carry a one-off rule before that domain exists.

What about audio?

Native audio generation is a route setting. Audio conditioning is an input dependency. The route catalog tells core which one applies, so the plan either shows a setting or a dependency node, not both by accident.

07 - Non-shot generations

The same dependency map can work for image generations later.

Shot video is the first place where this hurts, but it is not the only place where dependencies exist.

This avoids duplicating dependency logic in every purpose, while leaving the existing generation-lifecycle refactor to the follow-up plan.
The proposal is generic where it should be generic.

Dependency resolution, graph display, estimate aggregation, and execution ordering come from the shot-video plan. Refactoring older non-shot image generations onto a shared lifecycle belongs to plan 0043, after the shot-video dependency contract is proven.

AI Production pane

Shows compact model rows, route settings, and total estimate. It can show `$3.40`, `$3.40 + unpriced`, `$3.40 + pending`, or `Needs plan` based on plan state.

Preview Take Plan dialog

Shows the graph: reused references, planned generations, missing attachments, unpriced lines, individual estimates, and final video payload summary.

Agent access

Materialize the same graph into persisted specs, request approval for priced lines or explicit unpriced override, and run dependency steps before the final video.

09 - Completion checklist

Done means the same graph estimates, previews, and runs.

The checklist in the markdown plan tracks the work from design review through validation. This is the high-level version.

Engines

Route catalog complete

  • All six base models are explicit families.
  • Every supported intent has a route.
  • Each route declares inputs, settings, durations, and pricing.
  • Validation catches invalid fields before Studio says Ready.
Core

Dependency planning complete

  • Dependency map resolves reused, planned, missing, and unpriced lines.
  • Reference intent resolves cast, location, lookbook, audio, and video slots.
  • Total estimate includes dependency generation plus final video.
  • Partial and unpriced estimates are explicit, not silent dashes.
Product surfaces

Same plan everywhere

  • AI Production renders model rows and route setup from core.
  • Preview Take Plan dialog renders the dependency map.
  • CLI prints the same plan for the agent workflow.
  • Agent materializes and runs the same graph after approval or override.
Purposes

Purpose contracts complete

  • Shot frame image purposes can build and estimate specs.
  • Reference bundles resolve concrete purposes or attached media.
  • Shot video take purpose can build and estimate specs.
  • Existing purposes are used without copying their internals into shot video.
Run flow

Materialization complete

  • Dependency and final specs share a plan id.
  • Approval summary includes priced lines and unpriced overrides.
  • Dependency assets are generated or attached before final video.
  • Actual costs can be audited against the original plan.
Validation

Confidence complete

  • Engine route matrix is tested.
  • Core tests cover every supported model and intent.
  • Stale route settings cannot leak.
  • Studio renders all plan states correctly.
The full checklist lives in the markdown plan.

It includes design review items, engine catalog tasks, core planning tasks, generation-purpose tasks, materialization tasks, Studio, CLI, agent workflow tasks, and validation tasks.
